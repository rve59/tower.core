## 1. Analysis of Legacy LSL
The legacy TOWER-LSL focused on a **Static Layout Paradigm**. It combined three concerns into a single syntax:
1.  **Structure**: The geometric arrangement of rows and columns.
2.  **Components**: The specific leaf-node UI elements (Buttons, Inputs).
3.  **Binding**: The direct mapping to data models.

**Limitation**: To create a different product (e.g., moving from a Single-Entity view to a Fleet-Manager view), the developer had to copy-paste the entire LSL and modify the leaf-nodes, leading to code duplication and maintenance drift.

---

## 2. The LSL-MPL (Multi-Product Line) Paradigm
LSL-MPL introduces a **Decoupled Assembly** model. It separates the "Skeletal Frame" (TOWER-U) from the "Feature Content" (TOWER-F).

### 2.1 The Blueprint Token
The `BLUEPRINT` block is the primary entry point for a product. It defines the mapping between a **Dock ID** and a **Feature ID**.

```text
BLUEPRINT <ProductID>
  THEME <ThemeClass>
  MAP #<DockID> -> <FeatureID> (@context_binding)
```

### 2.2 Universal Regions (TOWER-U)
A `REGION` in LSL-MPL defines the structural "holes" in the application shell. 

```text
REGION <Name> <Layout> <Sizing>
  SLOT Icon #logo          // Static element
  DOCK #navigation         // Injection point resolved by the Blueprint
```

---

## 3. Syntax Specification

### 3.1 Structural Tokens
- **`REGION`**: Defines a structural container. Parameters: `[Name] [col|row] [fill|fixed|hug|N%] [bar|panel|none]`.
- **`STACK`**: A dynamic navigation container. Parameters: `#id @router.binding`.
- **`SLOT`**: A **Terminal UI Component**. Used for static elements hardcoded into the layout. Parameters: `[ComponentType] #[id]`.
- **`DOCK`**: A **Feature Injection Point**. A structural "socket" for TOWER-F assemblies. Content is resolved by the Blueprint. Parameters: `#[id]`.

### 3.2 Mapping Tokens
- **`MAP`**: Used in the Blueprint to assign a **TOWER-F Feature** to a **DOCK**.
- **`THEME`**: Injects a global CSS signature (e.g., `.theme-pro`, `.theme-core`).
- **`CONTEXT`**: Acts as the **Data Bridge** between the global platform and the feature. It specifies which global state (e.g., `@platform.active_account`, `@user.permissions`) is injected into the DOCK. This ensures that isolated features can react to global changes (like switching companies) without being hard-wired to the platform logic.

## 4. Example: Assembling TOWER-M (The BPA Product)

### The Blueprint (`product_m.lsl`)
```text
BLUEPRINT TOWER-M
  THEME .theme-pro
  CONTEXT @platform.active_account
  
  MAP #global_actions -> F_Account_Switcher
  MAP #navigation      -> F_Fleet_Navigator
  MAP #work_area       -> F_Risk_Heatmap
```

### 4.3 Feature Specifications (TOWER-F)
These are the definitions of the units that are "docked" into the layout.

```text
FEATURE F_Fleet_Navigator
  TYPE NavigationTree
  BIND @entities.list
  ATTR density="compact"
  EMIT CIDSelected -> platform.context

FEATURE F_Risk_Heatmap
  TYPE ComplianceGrid
  BIND @analytics.fleet_health
  ATTR refresh="5m"
  EMIT RowClick -> detail_view
```

### 4.4 The Assembled Universal Layout (`shell_u.lsl`)
This is the "Standard Chassis" used by both TOWER-M and TOWER-C.

```text
REGION TOWER-U_Shell col fill
  REGION 6 Header row bar
    SLOT Icon #tower-logo        // Static element
    DOCK #global_actions         // Dynamic Feature injection point
  
  REGION Body row fill
    REGION 15 Sidebar col panel
      DOCK #navigation           // Injects F_Fleet_Navigator (for M) 
                                 // or F_Linear_Nav (for C)
    
    REGION MainStage col fill
      DOCK #work_area            // Injects F_Risk_Heatmap (for M)
                                 // or F_Audit_Grid (for C)
```

## 5. Technical Implementation: React + Vite
1.  **Registry**: A `FeatureRegistry.ts` (Auto-generated by GAG) that maps `F_FeatureID` to a lazy-loaded React component. GAG maintains this file synchronously as features are defined in LSL-MPL.
2.  **Dock Component**: A `<Dock id="navigation" />` component performs a lookup in the active Blueprint context to retrieve the assigned TOWER-F feature.
3.  **Vite Build**: The `VITE_PRODUCT_TARGET` environment variable selects which `.lsl` blueprint is loaded as the default entry point.

## 6. Comparison Table
| Feature | Legacy LSL | LSL-MPL |
| :--- | :--- | :--- |
| **Philosophy** | "Build an App" | "Assemble a Product" |
| **Injection Point** | Hardcoded `SLOT` | Dynamic `DOCK` |
| **Layout Reuse** | Copy-Paste | 100% Shared (TOWER-U) |
| **Theming** | Inline Overrides | Global Build-Time Themes |
| **Complexity** | High (per product) | Low (Config-only) |
